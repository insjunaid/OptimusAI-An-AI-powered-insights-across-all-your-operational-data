# OptimusAI Internal Architecture Document

## Overview
This document outlines the internal data flow for the OptimusAI project.

## Vector Database Strategy
We use ChromaDB to store vector embeddings of uploaded operational documents. 
When a document is uploaded:
1. It is parsed by `PyPDF2` (if PDF) or standard text extractors.
2. The text is chunked using LangChain's `RecursiveCharacterTextSplitter`. Our chunk size is 1000 characters with an overlap of 200 characters to ensure context isn't lost between chunks.
3. The chunks are embedded using Google's `text-embedding-004` model.
4. These embeddings are stored locally in the `./chroma_db` directory on the backend server.

## Relational Data
We use SQLite for local development (`ops_intel.db`). This database tracks:
- Workspaces
- Uploaded file metadata
- Chat history logs
- Summaries generated by the AI

## Security
API Keys are not stored in the database. Instead, they are read from environment variables on server startup. The `GEMINI_API_KEY` is mandatory.
